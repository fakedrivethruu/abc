# M0.2 ‚Äî Store: schema, migrations, single-writer access layer

Depends on: M0.1. Reads: SPEC ¬ß3‚Äì¬ß5, ¬ß11; DECISIONS D4, D14.

## Objective

The durable heart: SQLite opened WAL/foreign-keys-on, migrations applied via
`PRAGMA user_version`, one writer connection owned by a store task, and a typed API covering
every query later milestones need ‚Äî fully unit-tested. No HTTP wiring yet beyond what M0.1
left.

## Deliverables

- `hub/migrations/0001_init.sql` ‚Äî exactly the DDL in SPEC ¬ß4 (copy it; do not redesign).
- `hub/src/store/mod.rs` (+ submodules as needed):
  - `Store::open(path)` ‚Äî creates parent dirs, sets `journal_mode=WAL`, `foreign_keys=ON`,
    applies pending migrations in filename order inside a transaction each, bumping
    `user_version`; refuses to open if `user_version` exceeds known migrations.
  - Single-writer pattern per D4 (tokio-rusqlite or an mpsc command task ‚Äî pick one, note
    the choice in PROGRESS). All public methods are `async` and take/return typed structs
    (`Session`, `Event`, `Gate`, `Instruction`) mirroring SPEC ¬ß4; timestamps RFC3339 UTC.
  - Sessions: `create` (mints uuid, short, token ‚Üí returns `(Session, plaintext_token)`;
    stores SHA-256 only), `get`, `get_by_short`, `list(filters)` per SPEC ¬ß6 query params,
    `set_status` (validates the ¬ß5 machine, writes `session_status` event atomically in the
    same transaction, stamps `ended_at` on terminal), `set_claude_session_id`, `set_summary`,
    `set_error`, `set_plan_path`, `set_workspace`, `authenticate(bearer) -> Option<(Session,
    Role)>` by hash lookup.
  - Events: `append`, `list_after(session_id, after_id, limit)`.
  - Gates: `create` (computes `expires_at` = now + timeout), `get`, `decide(id, decision,
    decided_by)` (only from `pending`, atomic, appends `gate_decided` event),
    `expire_due(now)` (sweeper query per ¬ß6), plus a tokio `Notify`/watch mechanism so
    long-pollers wake on decide.
  - Instructions: `create` (+`instruction_created` event), `take_undelivered(session_id)`
    (atomic stamp + `instruction_delivered` events).
- `hub/src/store/tests.rs` (or colocated `#[cfg(test)]`) using tempfile DBs.
- Wire `Store::open(HUB_DATA_DIR/hub.db)` into hub startup; auth middleware from M0.1
  extended: admin token OR `store.authenticate(bearer)` attaches `AuthCtx {Admin |
  Session{id, role}}` to the request. Routes still to come.

## Steps

1. Copy DDL into migration file. 2. Implement open/migrate + a test proving re-open is
idempotent and forward-only. 3. Implement each entity module test-first. 4. Status-machine
validation table with tests for every legal and one illegal transition per state. 5. Gate
decide/expire concurrency test (two racing decides ‚Üí exactly one wins, other gets
`already_decided` error). 6. Wire startup + auth. 7. `make check`; PROGRESS.

## Acceptance criteria

1. `make check` green; store tests cover: migration idempotency, session create/auth
   round-trip (wrong token ‚Üí None), every legal ¬ß5 transition + rejection of
   `executing‚Üíawaiting_approval`, event append/list_after ordering, gate expire sweep flips
   only overdue rows, racing gate decides, instructions delivered exactly once under two
   concurrent takers.
2. Fresh run of the hub creates `$HUB_DATA_DIR/hub.db`; `sqlite3 hub.db "PRAGMA
   user_version; .tables"` shows `1` and the four tables.
3. `curl` any `/v1/*` path with a made-up session bearer ‚Üí 401; with admin token ‚Üí 404
   (unrouted but authed) ‚Äî proves the AuthCtx path compiles end-to-end.
4. No `unwrap` outside tests (`grep -rn "unwrap()" hub/src --include=*.rs | grep -v tests`
   returns only test files / documented startup expects).

## Out of scope

HTTP routes beyond auth plumbing, spawning, notifications, sweeper scheduling loop (the
query lands here; the 5 s loop lands in M2.2 where gates first exist at runtime).
