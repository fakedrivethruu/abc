# M0.3 — Spawner trait, host spawner, provisioning, first live session

Depends on: M0.2. Reads: SPEC §2 (injected env), §6 (sessions endpoints), §7 (posttool),
§8 (provisioning + invocation), §10, §12; DECISIONS D5, D8, D10, D16.

## Objective

`POST /v1/sessions` with `mode=direct` produces a real agent run end to end: recipe
resolved, workspace provisioned per SPEC §8, `claude -p` spawned with hooks wired, events
flowing into the store via `agentry-hook posttool`, terminal status + summary recorded, and
a notification attempted. This is the milestone where the system first breathes.

## Deliverables

- `hub/src/recipes.rs`: read-side port of the legacy resolver — resolve name→
  `recipe.toml` via `AGENTRY_RECIPES` search path (colon-separated, same semantics as
  `src/recipe.rs`), parse name/description/repository/claude_md_path plus SPEC §10
  `[policy]` (absent ⇒ defaults: `default="allow"`, Write/Edit/MultiEdit/Bash = "gate",
  `gate_timeout_seconds=540`, `mcp.servers=[]`). List function for `GET /v1/recipes`.
- `hub/src/spawn/mod.rs`: `trait Spawner { async fn start_phase(&self, req: PhaseRequest)
  -> Result<PhaseHandle>; async fn stop(&self, session_id) -> Result<()>; }` where
  `PhaseRequest` carries session row, phase number, prompt, workspace paths, env map, and
  optional `resume_claude_session_id`; `PhaseHandle` resolves to `PhaseOutcome
  {claude_session_id, result_text, is_error, raw_json}`.
- `hub/src/spawn/host.rs` (`HostSpawner`): shallow clone, render `claude-settings.json` /
  `mcp.json` (0600) / `policy.json` exactly per SPEC §8+§10, copy recipe CLAUDE.md, write
  `.git/info/exclude` entries, build the SPEC §8 command line, spawn with cwd/env per SPEC,
  capture stdout/stderr, parse the output JSON, kill-on-stop via a process registry
  (session_id → child). Workspace and all artifacts persist after exit (nothing deleted).
- `hub/src/orchestrator.rs`: drives a session through §5 — for `mode=direct`: created →
  executing (phase 2 semantics: prompt passed through verbatim) → done/failed; writes
  `phase_started`/`phase_completed` events; on process exit stores summary/error; sends
  §12 `session_status` notifications (reqwest client with 3× backoff; failure ⇒
  `notify_failed` event, non-fatal). `mode=plan_first` returns 501 `not_implemented` until
  M2.1 — an explicit refusal, not a silent fallthrough.
- HTTP routes (SPEC §6): `POST /v1/sessions`, `GET /v1/sessions[?filters]`,
  `GET /v1/sessions/{id}`, `POST /v1/sessions/{id}/stop`, `GET+POST
  /v1/sessions/{id}/events`, `GET /v1/recipes`. Auth per §3 (session token may only touch
  its own events).
- `hook/`: implement `posttool` per SPEC §7 (stdin parse, 8 KiB digesting, POST event,
  network failure → stderr + exit 0). Other subcommands still loudly unimplemented.
- `hub/tests/session_direct.rs`: integration test with a `FakeSpawner` (test-only) proving
  the orchestrator/state/event/notify pipeline without invoking claude; plus a
  provisioning unit test asserting rendered files byte-match SPEC §8 templates for a fixture
  recipe.
- `scripts/e2e/00_direct_session.sh`: creates a throwaway local git repo, a temp recipe
  (policy: everything "allow" — gating isn't built yet), POSTs a session ("create
  HELLO.md containing the word hello, commit it"), polls to terminal, prints status,
  summary, event count, and `git -C <ws>/repo log --oneline`.

## Steps

1. recipes.rs with tests (search path, missing repo error, policy defaults). 2. Spawner
trait + provisioning renderer with byte-exact template tests. 3. HostSpawner (guard actual
`claude` invocation behind the e2e path; unit tests stop at command construction).
4. Orchestrator + FakeSpawner integration test. 5. Routes + auth scoping tests. 6. Hook
posttool + a test against a stub HTTP listener. 7. e2e script; run it live (SETUP A1–A4, A9).
8. `make check`; PROGRESS.

## Acceptance criteria

1. `make check` green including the new integration tests.
2. `scripts/e2e/00_direct_session.sh` (human-run ok): session reaches `done`;
   `HELLO.md` exists and is committed in the workspace clone; `GET
   /v1/sessions/{id}/events` shows ≥1 `tool_use` from the real hook plus
   phase/status events; `summary` is non-empty; workspace directory still exists.
3. `POST /v1/sessions/{id}/stop` during a long-running e2e variant → status `stopped`,
   process gone (verify via `ps`).
4. Session bearer for session A cannot POST events to session B (403/404 per §3).
5. `mode=plan_first` → 501 `not_implemented`.

## Out of scope

plan_first flow (M2.1), pretool/stop/session-end hooks, gateway, container spawner,
relay notifications actually rendering anywhere (RELAY_NOTIFY_URL may point at
`scripts/e2e` netcat — the hub only needs to attempt them).

## Stop and ask if

Claude Code's actual `-p --output-format json` output shape differs from SPEC §8's assumed
fields (`session_id`, `result`, `is_error`) — verify against a live run FIRST, then propose
a SPEC clarification if needed rather than guessing.
