# M1.3 — Recipe generation: prompt → constrained recipe on disk

Depends on: M0.3 (recipes module, claude invocation plumbing), M1.2. Reads: SPEC §6
(`POST /v1/recipes`), §10; DECISIONS D8, D15.

## Objective

A plain-language prompt becomes a reviewable, constrained agent identity: `POST /v1/recipes`
runs a one-shot `claude -p` with a meta-prompt that must emit a strict JSON shape, the hub
validates and writes `recipe.toml` + `CLAUDE.md` into the search path, and `/agent new`
exposes it from Slack with a run affordance. Generated recipes are ordinary files the legacy
CLI can read (D15) and a human can diff before ever running.

## Deliverables

- `hub/prompts/recipe_meta.md` — the meta-prompt (a real, versioned deliverable). Required
  content: role statement ("you design a constrained identity for a coding agent");
  the exact output JSON schema `{name (kebab-case, 3–30 chars), description (≤120 chars),
  claude_md (markdown string), policy {default, tools{}, gate_timeout_seconds,
  mcp{servers[]}}}` with SPEC §10 vocabulary; hard rules the generated CLAUDE.md must
  encode — scope discipline, "stop rather than guess", never fabricate data, an output
  contract section, and an escalation section ("when to stop and let the operator decide");
  conservatism rules for policy (writes gate by default; Bash gets "deny" unless the task
  clearly needs shell; mcp.servers only when the prompt mentions CRM/HubSpot); a worked
  input→output example.
- `hub/src/recipe_gen.rs`: invoke `claude -p` with the meta-prompt + user prompt,
  `--output-format json` and the structured-output flag (`--json-schema` with the schema
  above) so the reply is machine-parseable; validate: name pattern, uniqueness in the first
  search-path root (conflict ⇒ suffix `-2`, `-3`…), policy vocabulary against §10, CLAUDE.md
  non-empty and contains the output-contract heading. Then write
  `<first-search-root>/<name>/{recipe.toml, CLAUDE.md}` atomically (temp dir + rename).
  Any validation failure ⇒ 502 `recipe_generation_invalid` with the specific violation —
  never "fix up" the model's output silently beyond the defined name-suffixing.
- Route `POST /v1/recipes` per SPEC §6; also emit a `message` notification to the invoking
  thread is NOT possible (no session) — instead the relay handles the reply directly.
- Relay `/agent new -- <prompt…>`: ack, call the endpoint, then post (visible, threaded):
  name, description, the policy table rendered human-readably, recipe path, and a button
  `recipe_run:<name>` that pre-fills a `run` for it — clicking asks for the prompt via a
  plain follow-up instruction message ("reply with `/agent run <name> -- <prompt>`"; a
  modal is optional polish, not required).
- Tests: recipe_gen validation logic against fixture JSON blobs (valid, bad name, bad
  policy vocab, missing CLAUDE.md section, name collision) — the claude invocation itself
  is behind a trait so fixtures drive tests; one live e2e in
  `scripts/e2e/03_recipe_gen.sh` generating a recipe from "a careful agent that fixes
  typos in markdown docs of a given repo" and asserting `agentry recipes show <name>`
  (the LEGACY binary) prints it.

## Steps

1. Write the meta-prompt. 2. Trait-wrapped generator + validators, test-first. 3. Route +
relay command. 4. Live e2e incl. legacy-CLI compatibility check. 5. `make check`; PROGRESS.

## Acceptance criteria

1. `make check` green.
2. Live: `/agent new -- an agent that fixes typos in markdown docs` posts a recipe card;
   the files exist under `AGENTRY_RECIPES`; `cargo run --bin agentry -- recipes show
   <name>` (legacy) renders it; the generated policy gates Write/Edit and denies or gates
   Bash (conservatism rule observed).
3. Generating the same prompt twice yields `<name>` then `<name>-2` — no overwrite.
4. A fixture with `policy.tools.Write = "yolo"` → 502 naming the invalid value; nothing
   written to disk.
5. `GET /v1/recipes` includes the new recipe with `has_policy: true`.

## Out of scope

plan_first default switching (M2.1 flips `/agent new`'s follow-up guidance to plan mode),
recipe editing/deleting endpoints, embedding repository URLs the prompt didn't specify
(repository stays unset unless the prompt names one — `--repo` at run time covers it).

## Stop and ask if

Structured output (`--json-schema`) behaves differently live than documented — capture the
raw output, stop, and report rather than hand-parsing around it.
