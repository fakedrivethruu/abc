# M1.1 — Relay: Socket Mode connection, hub client, notify listener

Depends on: M0.1 (relay scaffold), M0.3 (a hub to talk to). Reads: SPEC §1, §2, §6, §12;
DECISIONS D1, D11; SETUP A5.

## Objective

The relay stands: connected to Slack over Socket Mode, able to call the hub with the admin
token, and able to receive hub notifications on `127.0.0.1:8791/notify` and render them into
the right channel/thread. Proven with a ping command and a curl-injected notification.

## Deliverables

- `relay/src/hub.ts`: typed hub client (fetch wrapper) — base URL + bearer from config,
  SPEC §12 error envelope surfaced as thrown `HubError{code,message}`; methods needed so
  far: `healthz()`, plus the generic `get/post` used by later milestones.
- `relay/src/slack.ts`: Bolt `App` in Socket Mode (`appToken`, `token`), started from
  `src/index.ts`; graceful shutdown on SIGINT/SIGTERM.
- `relay/src/notify.ts`: minimal HTTP listener on `RELAY_BIND` (Node `http` — no extra web
  framework) accepting `POST /notify`: verify `Authorization: Bearer $RELAY_TOKEN` (401
  otherwise), validate body against SPEC §12 union (reject unknown `type` with 400), then
  render: `session_status` → threaded text "`<short>` (<recipe>) → *<status>*" plus summary
  or error when present; `message` → the text verbatim; `plan_ready`/`gate_created` → for
  THIS milestone, render text-only placeholders noting "interactive buttons arrive in
  M2.1/M2.2" is NOT acceptable — instead: render them as plain text with the excerpt/summary
  and a note that no action is attached yet ONLY if those types are received; the hub cannot
  emit them before M2.x, so in practice this path is exercised by tests. Respond 200 only
  after Slack accepts the post; Slack failure → 502 (the hub's retry handles it).
- `relay/src/commands.ts`: register `/agent`; subcommand router; implement `ping` → hub
  `healthz()` → ephemeral reply "hub ok (vX) — relay ok". Unknown subcommand → ephemeral
  usage text listing available subcommands (grows in M1.2).
- Tests (vitest): notify auth (401/400/200 paths) with Slack client stubbed at the module
  boundary; hub client error-envelope parsing.

## Steps

1. hub.ts + tests. 2. Bolt app boot + `/agent ping`. 3. notify.ts + tests. 4. Manual live
check per acceptance. 5. `make check`; PROGRESS.

## Acceptance criteria

1. `make check` green (relay tests included).
2. Live (SETUP A5 done): `make dev-relay` logs a Socket Mode connection; `/agent ping` in
   the test channel returns the ephemeral hub-ok line while `make dev-hub` runs; with the
   hub down, `/agent ping` returns a clear "hub unreachable" ephemeral (no stack trace to
   the user, full error to relay logs).
3. `curl -X POST 127.0.0.1:8791/notify -H "Authorization: Bearer $RELAY_TOKEN" -d
   '{"type":"message","session":{"id":"t","short":"t","recipe_name":"t","status":"done",
   "summary":null,"error":null,"slack_channel_id":"<your channel>","slack_thread_ts":null},
   "text":"notify path live"}'` → the message appears in the channel; same curl with a bad
   token → 401 and nothing posted.
4. Relay logs never contain token values (grep the log for the token strings).

## Out of scope

All other subcommands (M1.2), Block Kit interactivity (M2.1/M2.2), any hub-side changes.

## Stop and ask if

Socket Mode won't connect with valid-looking tokens (likely a missing scope — list what
Slack's error says and stop), or Bolt requires a config field SPEC §2 doesn't define.
