# M1.4 — Email bridge — DEFERRED (do not start; DECISIONS D13)

This file exists to reserve the design so nothing built earlier forecloses it. Any agent
assigned this file by mistake: stop and confirm with a human first.

Shape when activated: a second thin relay speaking the SAME hub API and notification
contract (SPEC §6/§12 — already transport-agnostic). Inbound: IMAP polling or the owner's
existing inbox project (`INBOX_TOKEN` plumbing in `scripts/agentry-helpers.sh` is the
integration seed) parsing `agent: <command>` subjects/bodies into the identical command set
as Slack. Outbound: SMTP posts of the same rendered notifications. Approvals CANNOT be
buttons: use signed one-time action links served by the email relay (HMAC over
gate_id+decision+expiry with a relay-local secret), and per-recipe gate timeouts raised to
hours for email-paced humans (SPEC §10 max would need a human-approved SPEC change — noted
now so it isn't "discovered" later). Session links generalize cleanly: `linked_ticket`
already exists; an email thread id would occupy a new `linked_thread` column via a new
migration.

Interfaces already honoring this: notifications carry full session context (relay
statelessness, D11); gates are decided via a hub endpoint any relay can call; nothing in
WS1–WS3 may assume Slack is the only relay.
