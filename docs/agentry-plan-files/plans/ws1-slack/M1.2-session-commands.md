# M1.2 — Slack session commands: run, list, show, stop; thread ↔ session

Depends on: M1.1, M0.3. Reads: SPEC §5, §6, §12.

## Objective

The full session lifecycle is drivable from Slack: `/agent run` starts a `mode=direct`
session whose lifecycle notifications land as a thread under the invoking message, and
`list`/`show`/`stop` work against the durable store (including finished sessions — history
is visible, nothing disappears).

## Deliverables

- `relay/src/commands.ts` extended (each replies fast — ack within Slack's 3 s window,
  do hub work after ack, then post results):
  - `/agent run <recipe> [--repo <path-or-url>] [--ticket <id>] -- <prompt…>` — parse
    (prompt required; support the `--` separator so prompts may contain flags-looking
    text), post a visible "starting `<recipe>`…" message to the channel, then `POST
    /v1/sessions` with `mode=direct`, `slack_channel_id` = channel, `slack_thread_ts` = the
    ts of that starting message (so every subsequent hub notification threads under it),
    then update the starting message with the session short id. Hub errors (unknown recipe,
    no_repository) render as friendly ephemerals quoting the hub's message.
  - `/agent list [status]` — `GET /v1/sessions` (+filter), render a compact ephemeral
    table: short, recipe, status, age, thread permalink when links exist. Empty → say so.
  - `/agent show <short-or-id>` — full detail ephemeral: status, phase, repo, workspace
    path, summary/error, created/ended, counts from `GET events` (e.g. "14 tool uses").
  - `/agent stop <short-or-id>` — confirm via a small Block Kit confirm button
    (`action_id stop_confirm:<id>`) then `POST /v1/sessions/{id}/stop`; report result in
    thread.
- Hub: ensure `GET /v1/sessions?…` supports lookup by `short` (add `short=` filter — this
  is within SPEC §6's filter list intent; if you read it as an addition, propose the
  one-line SPEC clarification first) and that notifications from M0.3 include everything
  §12 requires — fix hub gaps found here rather than working around them in the relay.
- Rendering module `relay/src/render.ts`: one place mapping notification/status → Block Kit
  or text, unit-tested with snapshot-style assertions (status transitions, summary
  truncation at 3000 chars with "…", error rendering).
- `scripts/e2e/02_slack_direct.sh` documentation block (this one is human-driven): the
  exact Slack inputs and expected outputs for the acceptance run.

## Steps

1. Argument parser + tests (quoted prompts, `--` handling, bad input messages). 2. run
command wired end to end against dev hub. 3. list/show/stop + confirm flow. 4. render.ts
extraction + tests. 5. Live acceptance run. 6. `make check`; PROGRESS.

## Acceptance criteria

1. `make check` green.
2. Live: `/agent run <toy-recipe> -- create HELLO.md saying hello and commit it` → starting
   message appears, gains the short id, and the SAME thread receives `executing` and then
   `done` with the summary; the file is committed in the workspace.
3. `/agent list` after completion still shows the session (status done) — durability
   visible in Slack; `/agent show <short>` renders detail incl. tool-use count.
4. `/agent stop` on a deliberately long-running session (prompt: "run sleep 120 via Bash
   then …", recipe policy Bash=allow) transitions it to `stopped` in-thread within ~5 s.
5. Malformed commands (`/agent run`, `/agent show nope`) produce helpful ephemerals, never
   silence, never a crash in relay logs.

## Out of scope

`/agent new` (M1.3), plan approval UX (M2.1), gates (M2.2), `/agent tell`/`history` (WS3),
email.

## Stop and ask if

Slack's 3 s ack constraint forces an interaction pattern SPEC/D11 don't cover, or the
`short=` filter reading needs the SPEC clarification.
