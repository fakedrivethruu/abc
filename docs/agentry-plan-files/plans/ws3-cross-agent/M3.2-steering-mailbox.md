# M3.2 — Steering mailbox: mid-flight instructions reach running agents

Depends on: M2.2 (pretool hook already calls the instructions endpoint), M0.2 (store).
Reads: SPEC §6 (instructions), §7 (pretool piggyback, stop), §11, §12; DECISIONS D12
context (managers use this in M3.3).

## Objective

A human can redirect a running agent without killing it: `/agent tell <short> -- <text>`
lands in the instructions mailbox; the next tool call delivers it as `additionalContext`
through the pretool hook; and if the agent tries to STOP with instructions still
undelivered, the Stop hook blocks the stop and hands them over. Delivery is exactly-once
and fully audited. This is the mechanism the manager agent will drive in M3.3.

## Deliverables

- Hub instruction endpoints live per SPEC §6: `POST /v1/sessions/{id}/instructions`
  (admin, or manager session token — enforce §3's "only cross-session manager write" rule
  now, tested against a manufactured manager-role session even though manager UX arrives
  in M3.3), `GET …?undelivered=true` (own-session token; atomic take from M0.2). Events
  `instruction_created` / `instruction_delivered` flow.
- `hook/` `stop` implemented per SPEC §7: `stop_hook_active` guard first (exit 0 —
  loop protection), then fetch undelivered; any ⇒
  `{"decision":"block","reason":"Operator instruction(s): <joined texts>"}`; none ⇒ exit 0;
  hub unreachable ⇒ exit 0 with stderr note (a stop is not a write — failing open HERE is
  correct and deliberate; do not "fix" it to fail closed).
- Confirm pretool's piggyback (built in M2.2) now returns real texts — integration test:
  instruction created → next pretool stub-run includes it in `additionalContext` →
  second pretool run does NOT (exactly-once).
- Relay `/agent tell <short-or-id> -- <text…>`: resolve session, refuse (friendly) on
  terminal sessions, POST with `source: "slack:<user_id>"`, confirm in-thread ("queued —
  delivers at the agent's next tool call or stop"). Render `instruction_delivered` events
  in `/agent show` counts.
- Tests: endpoint auth matrix (worker token cannot instruct others; manager token can;
  admin can; own-session GET only), stop-hook unit tests (guard, block shape, empty, hub
  down), delivery exactly-once under concurrent pretool+stop takers (reuses M0.2's atomic
  take — prove it end-to-end at the HTTP layer).

## Steps

1. Endpoints + auth matrix tests. 2. Stop hook + tests. 3. Exactly-once HTTP-layer test.
4. Relay command. 5. Live acceptance. 6. `make check`; PROGRESS.

## Acceptance criteria

1. `make check` green.
2. Live: start a plan-approved session on a multi-step prompt; mid-execution `/agent tell
   <short> -- also create NOTES.md summarizing what you changed`; the run's remaining
   output references the instruction and NOTES.md exists in the workspace at `done`.
   Events show created → delivered exactly once.
3. Live stop-block: `tell` an agent that is about to finish (or a variant prompt that
   finishes fast); observe the Stop hook cycle — the transcript/summary shows it continued
   to handle the instruction rather than ending; no infinite loop (stop_hook_active guard
   verified by the run terminating).
4. `/agent tell` on a `done` session ⇒ friendly refusal, no row created.
5. Auth matrix tests demonstrate a worker token 403s instructing another session.

## Out of scope

Manager tooling and identity (M3.3), instruction editing/cancel, broadcast-to-many,
priority ordering (FIFO by id is the contract).

## Stop and ask if

Stop-hook blocking proves disruptive in practice (e.g. models loop despite the guard) —
capture a transcript and propose adjusting the reason text or delivery policy via SPEC
clarification instead of silently changing behavior.
