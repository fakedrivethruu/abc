# M3.3 — Manager control plane: an agent that reads the fleet and steers it

Depends on: M3.1 (history), M3.2 (instructions), M2.3 (MCP transport module). Reads:
SPEC §3 (manager role limits), §6 (/message), §13 (the contract this implements);
DECISIONS D12.

## Objective

The multi-agent capstone: a manager-role session whose only hands are `/mcp/control` —
list sessions, read events and history, send instructions, spawn workers — and which
provably CANNOT decide gates. A human converses with the manager in its Slack thread; the
manager observes live workers and injects mid-flight course corrections through the M3.2
mailbox. Oversight stays human; orchestration becomes delegable.

## Deliverables

- `hub/src/mcp/control.rs` on the M2.3 transport: the six §13 tools, each with a JSON
  input schema (typed deserialization — reject unknown/missing fields with actionable
  JSON-RPC errors, e.g. "unknown session_id; call agentry_list_sessions first"), MCP
  annotations (`readOnlyHint: true` on list/get/search; false on instruct/spawn), and
  concise descriptions written for a model consumer. Responses: compact JSON in the tool
  result text — sessions trimmed to the §6 list shape, events paginated via `after_id` +
  `limit` (default 50, max 200).
- Role enforcement: `/mcp/control` requires a manager-role session bearer (worker ⇒ 403,
  admin bearer ⇒ 403 too — humans use REST, not the agent plane; keep the surfaces
  distinct). `agentry_send_instruction` writes `source: "manager:<caller_session_id>"`.
  `agentry_spawn_session` creates `mode=direct, role=worker` sessions inheriting the
  manager's `slack_channel_id`/`slack_thread_ts` (its workers report into its thread) and
  returns `{id, short}`.
- The negative space, tested hard (D12): no gate tool exists in `tools/list`; a manager
  token on `POST /v1/gates/{id}/decision` ⇒ 403 `manager_cannot_decide` (test exists since
  M2.2 — extend it to a REAL manager session created through this milestone's flow);
  a manager instructing itself is allowed (harmless) but spawning managers is not
  (`role` is not a spawn parameter — workers only).
- Manager identity: `examples/recipes/manager/` in-repo (`recipe.toml` with
  `[policy] default="deny"` — its ONLY capability is the control plane;
  `policy.mcp.servers=[]` since provisioning grants `/mcp/control` by role per SPEC §8) and
  a `CLAUDE.md` instructing: observe before acting; prefer one clear instruction over many;
  never spawn more than N=3 workers without reporting back; summarize fleet state when
  asked; you cannot approve anything — say so when asked to.
- Provisioner: manager sessions get `mcp.json` with the `control` entry (SPEC §8), empty
  repo dir, no clone.
- Relay: `/agent manager start` (spawns role=manager, mode=direct with a standing prompt
  "await operator direction; introduce yourself briefly"); plain messages posted in the
  manager's thread are forwarded via `POST /v1/sessions/{id}/message` (SPEC §6 — implement
  the endpoint here: valid only when a `claude_session_id` exists and no phase is running,
  runs `--resume` with the text, 409 otherwise) and replies land back in-thread via the
  existing notification path (`message` type with the phase's result text).
- Tests: control tools against a seeded store (list/get/search/events shapes, pagination),
  role matrix (worker/admin/manager × control endpoint), spawn-inherits-links, /message
  state guard (409 while running), instruction source stamping.

## Steps

1. `/message` endpoint + tests. 2. control.rs tools + role enforcement, test-first.
3. Manager recipe + provisioning branch. 4. Relay manager UX. 5. Live acceptance.
6. `make check`; PROGRESS. 7. Final: walk the WHOLE PROGRESS table; every milestone `done`
or explicitly deferred ⇒ write the closing log entry.

## Acceptance criteria

1. `make check` green.
2. Live: `/agent manager start` → manager introduces itself in-thread. Ask it "what's
   running?" → it calls `agentry_list_sessions` (visible in its `mcp_call`-equivalent event
   trail / hub logs) and answers accurately against a fleet with one live worker.
3. Live steering: with a worker mid-execution, tell the manager "have the worker also add
   a CHANGELOG entry" → manager calls `agentry_send_instruction` → worker's mailbox
   delivers (M3.2 path) → CHANGELOG change appears in the worker's output. Instruction row
   shows `source: manager:<id>`.
4. Live spawn: ask the manager to "spin up the typo-fix recipe on the toy repo" → new
   worker appears in the manager's thread, runs, reports done.
5. D12 proof: ask the manager to approve a pending gate → it declines per its CLAUDE.md;
   force the API call in a test with its token → 403. `tools/list` on `/mcp/control`
   contains no gate tool.
6. Worker token on `/mcp/control` ⇒ 403; manager token on `/mcp/gateway` ⇒ 403.

## Out of scope

Autonomous manager loops/schedulers (the manager acts when spoken to or resumed — no
self-driving daemon), manager-spawning-managers, cross-workspace file access for the
manager, gate-decision delegation of any kind (permanent, D12).

## Stop and ask if

Resume-per-message latency makes manager conversation impractical (>~60 s p50 replies) —
report measurements; a persistent interactive mode is a SPEC/DECISIONS change, not an
improvisation.
