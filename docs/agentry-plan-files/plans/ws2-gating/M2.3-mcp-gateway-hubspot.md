# M2.3 — MCP gateway: hub-held HubSpot credentials, read-pass, write-gate, audit

Depends on: M2.2 (gate lifecycle + relay rendering), M0.3 (provisioning). Reads: SPEC §2
(HUBSPOT_*), §8 (mcp.json), §9 (the contract this milestone implements), §10 (policy.mcp),
§12; DECISIONS D6, D7, D14; SETUP A6–A7.

## Objective

Agents gain CRM hands without ever holding CRM credentials. The hub exposes
`POST /mcp/gateway` (stateless streamable-HTTP JSON-RPC per §9), proxies to a per-session
`@hubspot/mcp-server` stdio child authenticated from the hub's env, classifies every
`tools/call` read-vs-write via an allowlist that fails closed, gates writes through the
exact M2.2 human-decision machinery, records every call in the audit trail, and refuses to
start against anything but a sandbox portal. This is the production-write choke point.

## Deliverables

- `hub/src/mcp/transport.rs`: JSON-RPC request/response types + the axum handler shape for
  stateless streamable HTTP — one JSON response per POST, methods per §9 (`initialize`,
  `notifications/initialized` → 202 empty body, `ping`, `tools/list`, `tools/call`,
  else `-32601`). Shared by this milestone and M3.3. Consult the MCP spec while building
  (start at `https://modelcontextprotocol.io/sitemap.xml`, fetch pages with `.md` suffix) —
  match the current streamable-HTTP schema rather than guessing field names.
- `hub/src/mcp/upstream.rs`: per-session lazy child manager — spawn
  `npx -y @hubspot/mcp-server` with env `PRIVATE_APP_ACCESS_TOKEN=$HUBSPOT_PRIVATE_APP_TOKEN`
  (hub env only — never in any session env or file), run the stdio initialize handshake
  once, serialize requests per child (one in-flight request per child; queue behind a
  mutex — the proxy's JSON-RPC ids are per-client, map them), kill the child on session
  terminal status. Child crash mid-call ⇒ JSON-RPC error to the caller + `error` event;
  next call respawns lazily.
- Startup precondition (D7): if the gateway is enabled (token present) and
  `HUBSPOT_PORTAL_KIND != "sandbox"`, abort hub startup with a fatal log naming D7. Tested.
- `hub/config/gateway_tools.toml`: `read = ["*list*", "*get*", "*search*", "*read*"]` glob
  patterns — populate against the ACTUAL tool names from a live `tools/list` (record them
  in a comment); anything unmatched is a WRITE (fail closed, per §9). Unit tests: known
  read passes, known write gates, an invented tool name gates.
- `hub/src/mcp/gateway.rs`: the §9 `tools/call` flow — classification; reads forward +
  `mcp_call{direction:"read"}` event; writes create gate `kind=mcp` (summary: tool name +
  salient args like object type/id — derive from args, truncate sanely), wait on the M2.2
  decision path up to the session policy deadline; allow ⇒ forward + event with `gate_id,
  forwarded:true`; deny/expired ⇒ tool result `isError:true` with the §9 denial text.
  Idempotency dedupe per §9 (SHA-256 key, 10-minute window, in-memory map is acceptable —
  note volatility in a doc comment; the gate row is the durable record).
- Provisioner: include the `hubspot` entry in `mcp.json` (SPEC §8 shape) iff the recipe's
  `policy.mcp.servers` contains `"hubspot"`; auth = that session's bearer; worker role
  required by the endpoint (manager tokens 403 here).
- Tests: transport unit tests (each method, unknown method, bad JSON-RPC); gateway flow
  tests with a FAKE upstream child (a tiny stdio echo script in `hub/tests/fixtures/` —
  test-only, per CLAUDE.md rules) covering read-pass, write-gate-allow-forward,
  write-deny-isError, dedupe-hit, child-crash; classification tests.
- `scripts/e2e/06_hubspot_gate.sh` doc block (human-run, SETUP A6–A7): recipe `crm-demo`
  with `policy.mcp.servers=["hubspot"]`, Bash="deny".

## Steps

1. Fetch/read the MCP streamable-HTTP spec pages; write transport + tests. 2. Upstream
child manager against the fixture echo child. 3. Live `tools/list` against the real
@hubspot/mcp-server; fill gateway_tools.toml from reality. 4. Gateway flow + tests.
5. Startup precondition. 6. Provisioner wiring. 7. Optional local sanity: MCP Inspector
(`npx @modelcontextprotocol/inspector`) or `claude mcp add --transport http` against the
gateway with a session token. 8. Live e2e. 9. `make check`; PROGRESS.

## Acceptance criteria

1. `make check` green (fake-upstream suite included).
2. Startup with `HUBSPOT_PORTAL_KIND=production` ⇒ refuses to start, log names D7; with
   `sandbox` ⇒ starts.
3. Live e2e (sandbox portal): session prompt "list our CRM contacts and summarize" —
   completes with NO gate (read path), `mcp_call` read events present.
4. Live e2e: prompt "update contact <known sandbox contact> lifecycle stage to lead" —
   Slack gate appears naming the tool + contact; Allow ⇒ HubSpot sandbox shows the change;
   events show gate + forwarded write. Deny variant ⇒ `isError` denial reaches the agent
   (its output acknowledges it), contact unchanged in HubSpot.
5. Repeat-Allow within 10 min of an identical allowed write returns the cached result (one
   upstream execution — verify via a counter in the fake-upstream test AND absence of a
   duplicate change in the live sandbox).
6. `grep -r HUBSPOT hub/src` shows the token read in exactly one config site; provisioned
   session files and env contain no `HUBSPOT_*` (assert in a test).

## Out of scope

Other CRMs, OAuth/remote `mcp.hubspot.com`, prompts/resources MCP features (tools only),
production portal (forever out of scope for the agent — D7), rate limiting.

## Stop and ask if

Claude Code's HTTP MCP client rejects the stateless single-JSON responses (capture the
exact exchange). The approved fallback (D14) is `agentry-hook mcp-shim`: stdio bridge
provisioned as a command server in `mcp.json`, forwarding to the same hub endpoint —
implement it ONLY after reporting the transport failure, and keep the HTTP endpoint as the
canonical implementation behind it.
