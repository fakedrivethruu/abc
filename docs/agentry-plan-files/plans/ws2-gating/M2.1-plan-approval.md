# M2.1 — Plan approval: two-phase spawn with human gate on the critical path

Depends on: M0.3, M1.2. Reads: SPEC §5, §6 (approve/reject), §8 (phase prompts, --resume),
§12 (plan_ready); DECISIONS D9.

## Objective

`mode=plan_first` works end to end and becomes the default posture: phase 1 produces
PLAN.md and exits, the hub parks the session in `awaiting_approval` — durably, surviving a
hub restart — Slack renders the plan with Approve/Reject, and approval resumes the SAME
Claude session into execution. Rejection is a first-class terminal outcome.

## Deliverables

- Orchestrator: replace M0.3's 501 with the real flow — phase 1 uses the SPEC §8 plan
  wrapper prompt and `--output-format json`; on exit, read `<ws>/repo/PLAN.md` (missing or
  empty ⇒ status `failed`, error `plan_missing` — an agent that ignored its output contract
  is a failure, not something to paper over); copy the plan to
  `$HUB_DATA_DIR/plans/<id>.md`, set `plan_path`, persist `claude_session_id`, transition
  `planning → awaiting_approval`, emit the §12 `plan_ready` notification (3500-char
  excerpt + a note when truncated).
- Resumability across restarts: on hub startup, scan for sessions whose status implies a
  live process (`planning`, `executing`); since the process is gone, mark them `failed`
  (`error: "hub restarted during phase"`) — EXCEPT `awaiting_approval`, which needs no
  process and must survive untouched. Test this explicitly.
- `POST /v1/sessions/{id}/approve`: valid only in `awaiting_approval` (409 otherwise);
  transitions to `executing` and starts phase 2 via the spawner with
  `resume_claude_session_id` and the SPEC §8 execute prompt. `/reject` → `rejected`
  (terminal), notification sent. Both record `decided-by` semantics in the event payload
  (relay passes the Slack user id in the request body: add `{decided_by}` — this is within
  the endpoints as specified; the event payload carries it).
- Relay: `plan_ready` rendering with the excerpt in a code block + Approve/Reject buttons
  per §12 action ids; button handler calls the endpoint, then edits the message to show the
  outcome and who clicked (buttons removed after use — no double-approve UX); 409 from a
  race renders "already decided by <…>".
- Default flip: `/agent run` gains `--direct` to opt out; without it, runs are
  `mode=plan_first`. `/agent new`'s follow-up guidance now describes the plan flow.
- Tests: FakeSpawner-driven integration covering plan happy path, reject path,
  plan_missing failure, restart-during-awaiting_approval survival, restart-during-planning
  → failed, approve-in-wrong-state 409, double-approve race (one 200, one 409).
- `scripts/e2e/04_plan_approval.sh` doc block: the human Slack walkthrough, including one
  run where the hub is restarted between plan and approve.

## Steps

1. Orchestrator phase-1 path + plan capture, test-first with FakeSpawner. 2. Startup
reconciliation + tests. 3. approve/reject endpoints + race tests. 4. Relay rendering +
button handlers. 5. Default flip. 6. Live walkthrough incl. the restart case. 7. `make
check`; PROGRESS.

## Acceptance criteria

1. `make check` green with the new integration matrix.
2. Live: `/agent run <toy-recipe> -- refactor HELLO.md into a proper README` → thread shows
   plan excerpt with buttons; PLAN.md exists in the workspace; Approve → `executing` →
   `done`, and the execution demonstrably follows the plan (spot-check the diff);
   the phase-2 run visibly resumed (its transcript/summary references the plan, not a cold
   start).
3. Live reject path: buttons → Reject → `rejected` in-thread; `GET /v1/sessions/{id}` shows
   terminal state with `ended_at`.
4. Restart resilience live: kill the hub while `awaiting_approval`, restart, Approve still
   works and phase 2 runs.
5. A recipe/prompt engineered to skip PLAN.md ends `failed` with `plan_missing` — no silent
   fallthrough to execution.

## Out of scope

In-flight write gates (M2.2 — phase 2 here still runs under the session policy's existing
allow/deny, with gates becoming live next), plan editing/re-planning loops, timeout on
`awaiting_approval` (it waits indefinitely by design; the human can stop it).

## Stop and ask if

`--resume` does not behave as SPEC §8 assumes on a live check (verify early with a manual
two-command experiment before building on it).
