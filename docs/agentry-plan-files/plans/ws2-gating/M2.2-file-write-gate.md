# M2.2 — File/shell write gate: pretool hook, gate lifecycle, Slack Allow/Deny

Depends on: M2.1 (gates surface in the plan-approved execution phase), M0.2 (gate store).
Reads: SPEC §6 (gates), §7 (pretool), §10, §12 (gate_created); DECISIONS D6.

## Objective

Every policy-gated tool call pauses, physically, until a human decides in Slack — or is
denied. The enforcement point is the `agentry-hook pretool` PreToolUse hook: allow/deny/gate
per session policy, long-polling the hub for decisions, deny on any failure. This milestone
also brings the gate sweeper online. After this lands, "the agent is about to write —
Allow/Deny" is a lived Slack experience, not a design.

## Deliverables

- `hook/` `pretool` implemented exactly per SPEC §7, including: policy.json lookup with
  §10 defaults; gate creation with a human-grade `summary` derived from tool_input —
  `Write`/`Edit`/`MultiEdit`: relative file path + first-line/±120-char excerpt of new
  content; `Bash`: the command (truncated 200 chars); other gated tools: tool name + arg
  digest; deadline = policy `gate_timeout_seconds`, capped at 540; long-poll loop
  `?wait=25`; instruction piggyback via `additionalContext` (harmless no-op until M3.2
  populates the endpoint — the call is made now so the contract is exercised); every deny
  reason names its cause (`policy`, `operator`, `gate <id> expired`, `hub unreachable`).
  Unit tests via a stub HTTP server: allow, deny, expiry, hub-down ⇒ deny, non-gated tool
  ⇒ empty-output exit 0, oversized input digesting.
- Hub: `POST /v1/gates`, `GET /v1/gates/{id}?wait=`, `POST /v1/gates/{id}/decision` per
  SPEC §6 — decision path enforces §3 (admin only; BOTH session roles get 403, tested);
  long-poll implemented on the M0.2 notify mechanism (no busy loops); `gate_created`
  notification emitted; the 5 s sweeper task started at boot (expire ⇒ same wake path as a
  decision).
- Relay: `gate_created` rendering — summary in a code block, kind/tool badge, expiry
  countdown text, Allow/Deny buttons per §12 ids; handler posts the decision with the
  clicking user's id, edits the message to the outcome ("Allowed by @x" / "Denied" /
  "Expired — auto-denied" on a later status fetch), 409 race handled as in M2.1.
- Hub integration tests: gate create→decide wakes a waiting long-poll promptly (<1 s);
  sweeper expires an overdue gate and the long-poll returns `expired`; decision idempotency
  (second decision 409).
- `scripts/e2e/05_file_gate.sh` doc block: live matrix — allow, deny, timeout (use a test
  recipe with `gate_timeout_seconds = 30`).

## Steps

1. Hub gate endpoints + sweeper + long-poll, test-first. 2. Hook pretool + stub-server
tests. 3. Relay rendering/handlers. 4. Live matrix. 5. `make check`; PROGRESS.

## Acceptance criteria

1. `make check` green including the 403-for-session-tokens decision tests.
2. Live ALLOW: plan-approved session tries to Write; Slack shows the gate with the file
   path + excerpt; Allow → the file is written, session proceeds; events show `gate_created`
   then `gate_decided{allow}` then the `tool_use`.
3. Live DENY: Deny → the file is NOT written; the agent's subsequent output acknowledges
   the denial (the reason string reached the model); session continues or finishes without
   the write.
4. Live TIMEOUT (30 s test recipe): no click → gate `expired`, write blocked, Slack message
   self-updates on the next status render; `decided_by='system:timeout'` in the row.
5. Kill the hub mid-long-poll: the hook denies with `hub unreachable` (verify in events
   after restart via the agent's visible behavior + hook stderr in spawner logs) — fail
   closed demonstrated.
6. A `policy.tools.Bash="deny"` recipe: Bash attempt is denied locally with reason
   `denied by session policy` and NO gate row is created (deny ≠ gate).

## Out of scope

MCP/SaaS gating (M2.3 — the gateway, not this hook, owns that per D6), diff-quality
summaries beyond the excerpt rule, per-path policy granularity (tool-level only in §10).

## Stop and ask if

Real hook stdin shapes for Write/Edit/Bash differ from what the summary derivation assumes —
capture one real payload per tool first (a temporary debug log in a scratch session is
fine), then implement against reality.
